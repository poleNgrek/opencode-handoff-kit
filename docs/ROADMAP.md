# Roadmap

Deferred ideas for the kit. Nothing here is committed to a release. Treat each entry as a discussion starter — when an item becomes active work, graduate it to a `[Unreleased]` entry in [`CHANGELOG.md`](../CHANGELOG.md) (and an in-flight plan / merge request).

This file is intentionally lightweight. Add, remove, and reorder entries freely. Keep each item to a short paragraph: **idea**, **why**, **open questions or trade-offs**.

---

## Versioning

### `kitVersion` in `descriptor.json`

- **Idea:** Add an optional top-level `"kitVersion": "vX.Y.Z"` field that records the upstream tag a project was initialized or last upgraded against.
- **Why:** Today the relationship between a project's descriptor and the kit it was generated by is implicit. A `kitVersion` makes it grep-able, lets `bin/install-opencode-conductor.sh` warn on incompatible upgrades, and gives `/project-init` and refresh commands a hook for compatibility checks.
- **Open questions:** Set by `/project-init`? Auto-bumped by `bin/install-opencode-conductor.sh`? How does it interact with downstream forks (do forks use upstream tags, fork tags, or a composite `kitVersion: "vX.Y.Z+fork"`)?
- **Trade-offs:** One more thing to keep current vs better discoverability and tool-driven safety on upgrade paths.

### Engine: load `descriptor.json` from repo

- **Idea:** Allow [`tools/_opencode_engine.ts`](../tools/_opencode_engine.ts) to discover `descriptor.json` inside the repo (e.g. `<git-root>/.opencode-conductor/descriptor.json`) instead of always reading from `~/.config/opencode/projects/<projectKey>/descriptor.json`.
- **Why:** Closes the loop on **project-local mode**. Today durable data can live beside the repo, but the descriptor file itself still lives in OpenCode config — see [`docs/PATH_CONTRACT.md`](PATH_CONTRACT.md).
- **Open questions:** Precedence rules (repo wins? config wins? per-project override?). How do existing tooling commands (`opencode_bootstrap_branch`, `opencode_refresh_context`) discover the descriptor without a `projectKey` argument? Does this require a new `/project-discover` command?
- **Trade-offs:** Engine work and a `PATH_CONTRACT.md v2` section vs symmetry between control-plane and data-plane in project-local mode.

## Tooling

### CI / pre-commit guard for descriptor-driven paths

- **Idea:** A lightweight check (CI job or pre-commit hook) that fails when a command markdown or skill hard-codes a path that should be descriptor-driven (e.g. literal `~/.config/opencode/projects/...` outside of canonical examples).
- **Why:** Most regressions during this round of work were path-related. A guard rail keeps the *descriptor-driven paths* invariant honest as the kit grows.
- **Open questions:** Allowlist for canonical example blocks? Run on push, on MR, or both?

## Governance

### Stakeholder lenses, promoted from plan files

- **Idea:** The pre-execution review pattern used during plan files (SWE / LogOps / PM / Security lenses) is useful but lives only inside ad-hoc plan markdown today. Promote it to a reusable template — either a `/project-plan` command, a section in [`WORKFLOW.md`](../WORKFLOW.md), or a stub in `docs/`.
- **Why:** Standardizes the kind of review the user asked for during the optional-project-local-state plan, so it doesn't have to be reinvented every time.
- **Trade-offs:** More structure vs more friction for ad-hoc plans. Likely belongs as **opt-in** rather than mandatory.

## Knowledge

### Optional `/track-package` command

- **Idea:** A focused command that wraps `/scaffold-knowledge` discovery for a single named leaf, useful in CI / pre-commit pipelines or when a user knows exactly which package they want tracked next.
- **Why:** Discovery is interactive by design; some teams want a scriptable single-leaf path that does the same safety guardrails (root containment, symlink refusal, package-name regex) without prompting.
- **Open questions:** Should it accept `--area` and `--leaf` flags, or auto-resolve the leaf from cwd? Should it integrate with the legacy descriptor `sharedPackageKnowledge` section for non-default paths?
- **Trade-offs:** Extra surface area and two ways to do the same thing — vs unblocking automation.

### Removal of legacy `pseudoPackageDetection` object form

- **Idea:** Drop the legacy single-object form of `pseudoPackageDetection` once `descriptorSchemaVersion: 2` is universally adopted.
- **Why:** Reduces normalize-on-read branches in commands, simplifies tests, and lets the schema migration close out.
- **Open questions:** When? Tied to a major release. Does the install script proactively migrate?
- **Trade-offs:** A clean cut after one minor release reduces complexity at the cost of mild upgrade friction.

### Removal of legacy `<area>/packages/<pkg>` redirect stubs

- **Idea:** After one release cycle of source-tree-mirror convention, drop redirect stubs left at the legacy `<opencodeRoot>/<area>/packages/<pkg>/AGENTS.md` paths.
- **Why:** The convention path is the long-term home; redirect stubs are a migration aid only.
- **Open questions:** What is the deprecation window — one minor release or one major?

## Conventions for this file

- New entries go to the bottom of their section.
